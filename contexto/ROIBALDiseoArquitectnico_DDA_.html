Documento de Diseño Arquitectónico (DDA)

Proyecto: ROIBAL Asistente de Finanzas Personales (PWA)

Entorno Operativo: Windows 11 (Desarrollo) / Web & iOS/Android (Producción)

Estrategia de Despliegue: GitHub + Netlify (Hosting Frontend) + Supabase (Backend/Auth/DB)

1. Stack Tecnológico General

El ecosistema se divide de manera estricta en dos entornos (Desarrollo Local y Producción en la Nube), permitiendo trabajar de forma 100% autónoma y sin conexión a internet durante la etapa de programación.

  • Frontend (PWA): Flutter Web utilizando Dart. El compilado final generará recursos estáticos (HTML, CSS, JS y Service Workers) capaces de ser instalados como una aplicación nativa en iOS (Safari) y Android (Chrome). [1]
  • Entorno de Desarrollo (IDE): Visual Studio Code (VS Code) con las extensiones oficiales de Flutter y Dart instaladas. [1]
  • Gestión del Backend Local: Supabase CLI corriendo sobre Docker Desktop en Windows 11. Esto levantará de forma local el motor PostgreSQL, las APIs y el sistema de autenticación.
  • Administración de Base de Datos: DBeaver conectado al contenedor de PostgreSQL local a través del puerto expuesto 54322.
  • Alojamiento en Producción: Repositorio en GitHub con integración continua (CI/CD) conectado a Netlify para compilar y distribuir la PWA de forma automática en cada git push.

2. Características y Prestaciones Esperadas

🔑 Autenticación Centralizada (Google Auth)

  • Experiencia de Usuario: Botón único de "Iniciar sesión con Google". Sin contraseñas locales que recordar.
  • Mecanismo Técnico: Flujo administrado nativamente por el componente GoTrue de Supabase (OAuth2).
  • Persistencia: El token seguro de sesión (JWT) se almacena localmente en el navegador para evitar logueos repetitivos. El ID único otorgado por Google se mapeará automáticamente a la tabla interna auth.users de Supabase.

🛡️ Arquitectura Multiusuario Segura (Multi-tenant Compartido)

  • Estructura de Datos: Todos los usuarios convivirán en las mismas tablas físicas (expenses, categories, accounts).
  • Aislamiento: Cada tabla contendrá obligatoriamente una columna user_id vinculada como clave foránea a auth.users.id.
  • Garantía de Seguridad (RLS): Se activará Row-Level Security (RLS) en PostgreSQL. La base de datos denegará cualquier intento de lectura, actualización o eliminación si el user_id del registro no coincide exactamente con el auth.uid() extraído del token de la sesión activa del usuario.

📶 Estrategia Offline-First (Velocidad en la Calle)

  • Almacenamiento Local: Los gastos se guardarán instantáneamente en la memoria local del navegador (IndexedDB) utilizando abstracciones de Flutter como Hive o Isar. La interfaz responderá en milisegundos sin esperar confirmación de internet. [1]
  • Sincronización: Un proceso en segundo plano (Service Worker) detectará la recuperación de la conexión web y empujará en lotes (batch) los gastos locales hacia Supabase.

📸 Lectura Inteligente de Tickets (OCR Local)

  • Captura: Integración con la cámara del dispositivo móvil mediante la API web estándar (gestionada por el paquete image_picker de Flutter). [1]
  • Procesamiento de Imagen: Extracción de texto mediante Inteligencia Artificial corriendo en el propio dispositivo con la librería google_mlkit_text_recognition. Al ejecutarse localmente, no genera costos por consumo de APIs externas (como Google Cloud Vision).
  • Parseo de Datos: Algoritmos basados en expresiones regulares (Regex) y heurística para buscar palabras clave (como "TOTAL", "Neto", "$") y autocompletar el monto, la fecha y sugerir la categoría del gasto.
